# ProductsApi — Backend

A REST API  built using .NET 10 and MS SQL Server . The backend serves product and category data to a React Native mobile app and runs fully on your local machine.

| Technology | Version | Purpose |
|---|---|---|
| .NET | 10 LTS | Backend framework |
| ASP.NET Core Web API | 10 | REST API |
| Entity Framework Core | Latest | ORM / database access |
| MS SQL Server | Any local instance | Database |

| Swagger / Swashbuckle | Latest | API documentation and testing |

## Architecture Overview

I chose Clean Architecture because it keeps the business logic completely independent of frameworks, databases, and external concerns. Here is how the layers depend on each other:

The Domain layer is the core  it contains only plain C# classes with no NuGet dependencies at all. This means if I ever wanted to swap EF Core for Dapper, or SQL Server for PostgreSQL, I would only touch the Infrastructure layer and nothing else would break.

Each feature is handled by a dedicated Use Case class in the Application layer instead of putting all the logic inside controllers. This makes each piece of functionality easy to read, test, and change independently.

---

## Database Design

I used a model-first approach with EF Core. I defined the entities in C# and ran migrations to generate the tables. The database has two main tables:

**Categories** — stores product categories with a unique slug field used for URL-friendly filtering (e.g. `electronics`, `clothing`).

**Products** — stores product information with a foreign key to Categories. Each product has a title, description, price, thumbnail URL, and a created timestamp that defaults to UTC time on the database side.

The relationships are: one Category has many Products. When a Category is deleted, the delete is restricted — meaning you cannot delete a category that still has products attached to it.

Seed data is included in the migration so the database comes pre-populated with 4 categories and 6 products the moment you run `Update-Database`.

## API Endpoints

| Method | Endpoint | Description |
|---|---|---|
| GET | `/api/products` | Returns all products |
| GET | `/api/products/{id}` | Returns a single product by ID |
| GET | `/api/products/search?q={query}` | Searches products by title or description |
| GET | `/api/categories` | Returns all categories |
| GET | `/api/categories/{slug}/products` | Returns products filtered by category slug |

All endpoints return JSON. Errors return a consistent `{ "message": "..." }` format — internal exception details are never exposed to the client.

## Setup Instructions

### 1. Clone the repository

```bash
git clone https://github.com/your-username/ProductsApi.git
cd ProductsApi
```

### 2. Open in Visual Studio

Open `ProductsApi.sln` in Visual Studio 2022. Wait for NuGet packages to restore automatically. If they don't, right-click the solution → **Restore NuGet Packages**.

### 3. Configure the connection string

Open `src/ProductsApi.Api/appsettings.json` and update the connection string to match your local SQL Server instance:

```json
{
  "ConnectionStrings": {
    "Default": "Server=YOUR_SERVER_NAME;Database=ProductsDb;Trusted_Connection=True;TrustServerCertificate=True;"
  }
}
```

Replace `YOUR_SERVER_NAME` with your actual SQL Server instance name. To find it, open SSMS — the server name shown in the connection dialog is exactly what you need. Common values are:

```
localhost
.\SQLEXPRESS
LAPTOP-NAME\SQLEXPRESS
(localdb)\MSSQLLocalDB
```

### 4. Run the database migration

Open **Package Manager Console** in Visual Studio (`Tools → NuGet Package Manager → Package Manager Console`) and run:

```powershell
Update-Database -Project ProductsApi.Infrastructure -StartupProject ProductsApi.Api
```

This creates the `ProductsDb` database, generates all tables, and inserts the seed data automatically. You can verify it worked by opening SSMS, refreshing the Databases folder, and checking that `ProductsDb` appears with the `Categories` and `Products` tables inside it.

### 5. Run the API

Press **F5** in Visual Studio or set `ProductsApi.Api` as the startup project and click Run. The browser will open Swagger UI automatically at:

```
https://localhost:5001/swagger
```

All 5 endpoints are listed there and ready to test.

## Testing the API

Once running, open Swagger UI and try these requests in order:

1. `GET /api/categories` — should return 4 categories
2. `GET /api/products` — should return 6 products
3. `GET /api/products/1` — should return the Wireless Headphones detail
4. `GET /api/products/search?q=watch` — should return the Smart Watch
5. `GET /api/categories/electronics/products` — should return 3 electronics products
